# 매칭엔진 성능 개선

## 배경

코인 거래소의 핵심은 매수/매도 주문을 짝지어 체결하는 **매칭엔진**이다.
초기에는 빠른 구현을 위해 호가창(OrderBook)을 RDB(MySQL)에 저장하고, Spring `@Scheduled`로 일정 주기마다 매칭하는 방식으로 만들었다.

이후 Redis로 다음과 같이 발전시켰다.

1. **DB → Redis 마이그레이션** — 호가창을 Redis Hash로 옮김
2. **직렬화 의심 → HASH 직접 사용** — JSON 대신 Hash 필드 매핑. 개선 미미
3. **진짜 원인 발견** — 매칭 한 번에 Redis와 6~8회 통신. 네트워크 왕복이 누적
4. **Lua script 적용** — 매칭 로직 전체를 한 스크립트로 묶어 1회 통신으로 압축

자세한 시행착오는 [블로그 글](https://velog.io/@devinsight/Redis는-과연-무조건-빠를까)에 정리해두었다.
글의 결론은 다음과 같았다.

> *"무조건적으로 좋다고 하는 기술을 쓰는 것보다 수치화를 통해 개선해나가는 것이 얼마나 중요한지 알 수 있었다."*

다만 그 측정은 모두 **자원 제한이 없는 환경**에서 진행되어, "DB가 무너질 만큼의 트래픽"을 노트북에서 만들지 못했다.
당시 *"대용량 트래픽에서는 Redis가 DB보다 유리할 것이다"* 라는 가설을 세웠지만, 자원 제한 없는 환경에서 측정한 수치만으론 이를 증명하지 못했다.

본 문서는 그 가설을 **자원 제한 환경에서 정량 검증**한다.

## 측정 환경

대규모 트래픽 환경을 노트북에서 모사하기 위해, **저장소(MySQL/Redis)에 자원 제한**을 걸었다.

| 컨테이너 | CPU | Memory | 비고 |
|---|---|---|---|
| WAS (Spring Boot) | 제한 없음 | 제한 없음 | 본 실험의 변수 아님 |
| MySQL | 0.3 | 512M | 측정 대상 |
| Redis | 0.3 | 512M | **MySQL과 동일** — 공정 비교 |
| RabbitMQ, Prometheus, Grafana | 제한 없음 | 제한 없음 | 측정 인프라 |

> WAS를 묶지 않은 이유: 실험 변수는 *매칭엔진의 storage 종류*이지 WAS 처리량이 아니다.
> WAS를 같이 묶으면 그게 새 변수가 되어 비교가 흐려진다.

부하 도구는 K6, 메트릭은 Spring Actuator → Prometheus → Grafana로 수집했다.

### K6 시나리오

```js
stages: [
    { duration: '10s', target: <VU> },   // ramp-up
    { duration: '60s', target: <VU> },   // 유지
    { duration: '10s', target: 0 },      // ramp-down
]
```

- VU 200 / 500 두 단계로 측정
- VU id 짝/홀로 매수/매도 분기 (50:50)
- 가격 9000~11000 random (매칭 성립 가능성 확보)
- 수량 1~5 random
- 각 VU 요청 간격 0.1초

매칭엔진 모드는 application.yml의 `matching.engine` 프로퍼티로 토글 (`db` / `redis`).

### 해석

- **per-trade 매칭 자체는 빠르다** (1ms). 매칭 알고리즘에는 문제 없음.
- VU를 200 → 500으로 2.5배 늘렸는데 **RPS는 18%만 증가**. WAS가 throughput 천장에 도달.
- **에러는 0건** — 시스템은 살아있지만, 응답이 심하게 느려짐 (p95 876ms → 1.8s).
- **매칭 RPS가 오히려 폭락 (45 → 5 ops/s)** — 매칭엔진 스레드가 HikariCP 풀에서 사용자 스레드와 경쟁하다 밀림. *인입 처리하느라 매칭이 굶주림*.
- **HikariCP pending이 190까지 폭증** — 풀(10개)에 비해 명백히 부족.

> *"DB의 한계"는 매칭 알고리즘이 아니라 **커넥션 풀 + 폴링 주기가 만드는 throughput 천장**에서 온다.*

### 해석

- **200 VU**: Redis가 DB보다 약간 빠름 (RPS +21%, p95 -33%). 호가창 저장이 DB에서 빠진 만큼 DB 압력이 약간 덜어진 효과.

## 분석: 왜 Redis가 명확한 우위가 아니었나

### 1) HikariCP가 진짜 병목

DB 모드와 Redis 모드 모두 HikariCP pending이 비슷한 수준 (140~190).
**Redis 모드에서는 매칭엔진이 HikariCP를 사용하지 않는데도 풀이 똑같이 막힌다는 건, 풀의 진짜 소비자가 매칭엔진이 아니라는 뜻이다.**

실제 풀 소비자:
- HTTP `/api/orders/*` (orders INSERT)
- WalletService listener (wallet UPDATE — 매수 시 잔고 차감)
- TradeService listener (trade INSERT)
- OrderMatchedEventListener (wallet/coin_wallet UPDATE — 체결 후 정산)

매칭엔진은 전체 DB 사용의 일부이지, 지배적 소비자가 아니다.
→ 매칭엔진을 Redis로 옮겨도 시스템 전체 throughput에는 큰 변화가 없음.

### 2) Redis도 자원 제한에 걸림

500 VU에서 Redis가 더 느려진 이유:
- 500 동시 요청이 동시에 호가창 저장(`HMSET + 2x SADD`)을 Redis로 폭격
- Redis는 단일 스레드 이벤트 루프 → CPU 0.3에서 명령 큐가 쌓임
- DB(MySQL)는 다중 스레드 + 메모리 캐시로 동시 처리에 더 강함

→ 공정 비교를 위해 Redis와 MySQL 자원을 동등하게 묶었더니, 오히려 Redis가 *동시성에 약한* 면이 부각됨.

### 3) 측정의 한계 — 매칭엔진 단독 성능은 알 수 없음

지금 측정값은 **시스템 전체 throughput**이지 **매칭엔진 단독 throughput이 아니다.**

매칭 RPS는 다음 파이프라인의 *마지막 단계* 결과물이다.

```
HTTP order → wallet 처리 → 호가창 저장 → 매칭 트리거 → trade 기록 → 정산
```

앞단(HTTP/wallet)이 막히면 매칭 입력 자체가 늦어진다. 그래서 매칭 RPS 수치만으로는:
- 매칭엔진이 느린 것인지
- 매칭엔진에 들어갈 입력이 늦는 것인지

구분할 수 없다.

매칭엔진 단독 성능을 측정하려면 매칭엔진을 *isolate* 해야 한다 (호가창에 직접 데이터를 채워 넣고 매칭만 트리거하는 등). 그건 본 글의 범위를 벗어난다.

### 4) 매칭엔진은 이미 *논리적*으로 분리되어 있음

오해를 정리하면, 매칭엔진은 *실행 경로상* 이미 분리되어 있다.

| 분리 종류 | 분리됨? |
|---|---|
| 코드 경로 (HTTP request path) | ✅ |
| 실행 스레드 (Spring `@Scheduled`) | ✅ |
| 도메인 패키지 | ✅ |
| JVM 프로세스 | ❌ |
| **HikariCP 풀** | ❌ |
| Redis 클라이언트 풀 | ❌ |
| GC / 메모리 | ❌ |

문제는 *논리적 분리*가 아니라 **자원(특히 HikariCP) 공유**다.
그리고 매칭엔진은 풀의 지배적 소비자가 아니므로, 매칭엔진을 별도 프로세스로 떼는 것만으로는 시스템 throughput이 크게 좋아지지 않는다.

## 결론

당초 가설 *"대용량 트래픽에서는 Redis가 DB보다 유리할 것이다"*는 본 측정으로 증명되지 않았다.

- 같은 자원 조건에서 두 모드의 시스템 throughput은 비슷함 (200 VU에선 Redis가 약간 우위, 500 VU에선 DB가 약간 우위).
- 시스템의 진짜 병목은 매칭엔진이 아니라 wallet/trade의 DB 사용. 매칭엔진을 어디 두느냐만 바꿔서는 시스템 throughput이 크게 변하지 않음.
- 매칭엔진 단독 성능 비교는 본 측정으로는 알 수 없음 — *isolation 측정*이 별도로 필요.

*"Redis가 무조건 빠르다"* 는 직관이 시스템 단위에서는 깨질 수 있음을 수치로 확인했다.
