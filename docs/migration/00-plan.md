# MSA 분리 챕터 — 작업 계획

## 왜 이 챕터를 시작했는가

이 프로젝트는 처음부터 **대용량 트래픽 환경에서 동작하는 거래소 백엔드** 를 목표로 했다.
지금까지의 작업 — EDA 도입, SAGA 패턴, Redis 매칭엔진 — 은 모두 그 목표 위에서 진행한 것들이고, 단일 프로세스 안에서 할 수 있는 개선은 어느 정도 다 거쳤다.

그 다음으로 자연스럽게 마주친 질문이 *"단일 프로세스 한계를 넘어서려면 무엇이 필요한가"* 였다.

지금 구조는 도메인별로 패키지가 잘 나뉘어 있고 RabbitMQ 이벤트로 통신하지만, 결국 **단일 jar / 단일 프로세스 / 단일 DB** 에서 돌아간다.
이 상태에서는 어떤 도메인이 트래픽 폭주를 만나도 다른 도메인이 같은 자원(JVM, 커넥션 풀, DB)을 공유하기 때문에 같이 영향을 받고, 도메인 단위로 독립적으로 스케일하지도 못한다.

따라서 다음 챕터의 주제는 **이벤트 기반 모듈러 모놀리스를 진짜 분산 MSA로 발전시키는 것**으로 잡았다.
도메인을 별도 프로세스로 떼어내고, 자원을 분리하고, 메시징을 Kafka로 옮기고, 오토스케일링까지 붙이는 단계적인 작업이다.

## 챕터의 목표

끝나고 나면:
- 도메인별로 프로세스/자원이 독립되어 있어야 한다
- 메시징 백본이 RabbitMQ에서 Kafka로 이동해 있어야 한다
- 컨테이너 오케스트레이션 + 커스텀 메트릭 기반 오토스케일링이 동작해야 한다
- 분산 환경에서 도메인별 단독 성능 측정이 가능해야 한다

## 단계

각 단계는 한 단계씩 끝내고 회귀 검증을 거친 뒤 다음으로 넘어간다.
완성된 단계는 단계별 docs로 정리한다.

1. **멀티모듈 분리** — 단일 프로세스를 유지한 채 Gradle 모듈로만 자른다. 도메인 경계를 컴파일러로 강제. *([01-multi-module.md](./01-multi-module.md) — 완료)*
2. **프로세스 분리** — 도메인 모듈을 별도 Spring Boot 앱으로 분리. Gradle 의존성을 네트워크 의존(REST/이벤트)으로 변환.
3. **DB schema/인스턴스 분리** — 도메인별 schema 또는 DB 분리. 자원 공유로 인한 간접 결합을 끊는다.
4. **인증 처리** — gateway 도입 또는 각 서비스에서 JWT 검증.
5. **Kafka 마이그레이션** — RabbitMQ를 Kafka로 교체. 파티션 키 = coinId로 코인별 순서 보장 + 병렬화.
6. **k8s + HPA** — 매니페스트 작성, 커스텀 메트릭(Prometheus) 기반 오토스케일링.
7. **MSA 운영 이슈** — 회로차단기(Resilience4j), 분산 트레이싱(Zipkin/Jaeger), 헬스체크.
8. **부하 측정** — 분산 환경에서의 처리량/병목/스케일링 동작 측정.

## 미리 알고 있는 어려움

- *분산 SAGA의 망 위 동작 검증* — 메시지 중복/유실/순서 실전. 멱등성 점검이 단계 5 이후 본격적으로 필요해진다.
- *로컬 개발 경험 악화* — 5~6개 서비스 + Kafka + k8s를 노트북에서 돌리는 비용이 크다. kind/minikube/k3s 중 선택해야 한다.
- *HPA 메트릭 선택* — 단순 CPU 기준은 매칭엔진 같은 워크로드에 안 맞을 가능성이 크다. Prometheus → Custom Metrics Adapter 셋업이 단계 6의 핵심.
- *이벤트 스키마 진화* — 서비스가 독립 배포된다는 건 이벤트 DTO 호환성을 깨뜨리지 말아야 한다는 것. Avro/Protobuf 도입 여부는 단계 5에서 다시 결정한다.

## 진행 방식

- 단계마다 작업 노트를 짧게 남긴다 (왜 했는가 + 무엇을 했는가 중심).
- 정량 측정이 의미 있는 단계(8단계)에서 별도로 측정 docs를 작성한다.
- 챕터 전체가 끝나면 단계별 노트를 재료로 종합 회고를 한 번 더 정리한다.
