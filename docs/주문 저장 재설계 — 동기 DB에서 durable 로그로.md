# 주문 저장 재설계 — 동기 DB에서 durable 로그로

## 배경

이 프로젝트는 대학생 때 만든 것을 다시 보며 마이그레이션하고 있다.
주문 흐름은 그때 만든 구조 그대로였다 — 주문을 DB에 동기로 저장하고, 그 결과(auto-increment id)를 받아야 다음으로 넘어갔다.

수평 확장(HPA)을 붙이기 전에 무엇이 병목인지부터 측정했고, 그 병목이 이 재설계의 출발점이 됐다.
과정은 측정 → 개선 → 새 한계 측정 → 개선의 반복이다.

## 1. 병목 측정 — 매칭이 아니라 동기 DB 쓰기

주문 흐름의 각 구간에 타이머를 붙여 시간을 쟀다.

| 구간 | 평균 | 비중 |
|---|---|---|
| createOrder (DB insert) | ~2,900 ms | 76% |
| debit (HTTP → funds) | ~600 ms | ~20% |
| match (매칭 연산, Redis Lua) | ~4 ms | 0.1% |

거래소니까 매칭이 가장 무거울 거라 생각하기 쉬운데, 매칭 연산은 4ms로 전체의 0.1%였다.
시간의 대부분은 주문을 DB에 저장하는 createOrder였다.
INSERT 한 줄이 2.9초인 건, 200 동시 요청이 커넥션 풀 10개를 두고 경쟁하며 커넥션 대기가 누적된 결과다.

코드를 다시 보니 두 가지가 있었다.
1. 매칭 Lua는 Redis OrderBook만 읽고 쓴다. DB를 건드리지 않는다.
2. OrderBook(Redis)이 coinId·price·remainingAmount 등 작업 데이터를 이미 다 갖고 있다.

주문의 진실은 이미 Redis에 있었고, DB의 Order는 그 결과를 기록하는 사본에 가까웠다.
사본을 만드느라 hot path에서 동기로 기다리고 있던 셈이다.

## 2. 1차 개선 — Redis 우선, DB는 비동기 사본

DB 쓰기를 hot path에서 들어냈다.
id는 Redis INCR로 채번하고, OrderBook을 Redis에 등록한 뒤, DB 반영은 인메모리 이벤트(@Async)로 미뤘다.
돈이 걸린 debit/settle(동기 RPC)은 즉시 정합성이 필요해 hot path에 남겼다.

| | 이전 (동기 DB) | 1차 (비동기 projection) |
|---|---|---|
| 처리량 | ~50/s | 100/s |
| p95 | 4~8s | 2.8s |

처리량이 2배가 됐다.
같은 DB 쓰기인데, 200개 요청이 커넥션 풀을 두고 경쟁하며 하던 것을 단일 스레드가 경쟁 없이 순차로 처리하니 오히려 효율적이었다.

절대 수치(100/s)는 로컬 환경 천장 안이라 그대로 의미를 갖긴 어렵고([주문 흐름 병목 측정](주문%20흐름%20병목%20측정.md) 참고), 구조 변경 전후의 상대 비교(2배)가 유효하다.

## 3. 유실 재현 — 인메모리의 대가

1차 구조는 구멍을 남겼다. projection 큐가 인메모리라, trading이 크래시하면 아직 DB에 반영 안 된 주문이 유실된다.
"유실될 수 있다"는 논리만으로 넘어가지 않고, 실제로 얼마나 잃는지 재현했다.

projection이 밀린 상태에서 크래시하도록, trading-db를 `docker pause`로 멈춰 큐를 쌓았다.
그 상태로 800건을 접수시키고 trading을 하드킬(SIGKILL)한 뒤 재시작했다.

접수 800건, DB 반영 0건. **800건이 인메모리 큐와 함께 통째로 사라졌다.**
1차 구조에는 이를 복구할 수단이 없다.

## 4. 2차 개선 — durable 로그를 진실로

주문이 일어났다는 사실을 디스크에 남는 로그(Kafka)에 먼저 적고, 그 다음에 나머지를 진행하게 했다.

```
[hot path]
  debitKrw (RPC, 돈)
  id = Redis INCR
  OrderPlaced → Kafka append (동기, 저장 확인 대기)  ← 여기서 돌아오면 durable
  OrderBook → Redis (매칭용 작업 상태)
  match (Redis)
  체결당 settle (RPC, 돈) + TradeExecuted → Kafka append (동기)
  응답
       ↓
[컨슈머 — order-projection 그룹]
  로그 → DB 반영 (Order/Trade/fill), 반영 커밋 후 offset 커밋
```

역할이 세 층으로 갈렸다.
1. Kafka 로그 = 진실. durable한 원본.
2. Redis OrderBook = 작업 상태. 매칭이 도는 곳.
3. DB = 조회용 사본. 컨슈머가 로그를 읽어 반영하는 projection.

두 가지가 짝을 이룬다.
- **동기 append.** `send().get()`으로 저장 확인까지 기다린다. 여기서 돌아오면 프로세스가 죽어도 로그에 남는다. 비동기로 보내면 "보냈는데 저장 전 크래시" 윈도우가 남아 1차의 구멍과 같아진다.
- **dedup.** 컨슈머는 DB 반영을 커밋한 뒤 offset을 커밋한다. 그 사이 크래시하면 재시작한 컨슈머가 같은 이벤트를 다시 읽는다(at-least-once). 이벤트마다 eventId를 두고 processed_event에 마커를 남겨 중복 반영을 막았다.

5단계에서 알림용으로는 과하다며 되돌렸던 Kafka와 dedup이, 주문 이벤트 소싱에서는 필요해 다시 들어왔다.
되돌린 판단이 틀린 게 아니라, 알림에는 과했고 주문 로그에는 맞는 적재적소의 문제였다.

## 5. 유실 검증 + 비용

3번과 똑같은 조건(DB pause → 800건 접수 → 하드킬 → 재시작)으로 2차 구조를 측정했다.

| | 접수(200) | DB 반영 | 유실 |
|---|---|---|---|
| 1차 (인메모리) | 800 | 0 | **800** |
| 2차 (durable 로그) | 800 | 800 | **0** |

같은 크래시에 800 대 0이다.
2차는 접수 시점에 Kafka에 durable append됐으니, 재시작한 컨슈머가 로그의 offset부터 재개해 800건을 전부 복구했다.

내구성은 공짜가 아니었다.

| | 1차 (인메모리) | 2차 (durable 로그) |
|---|---|---|
| 처리량 | 100/s | 54/s |
| 크래시 유실 | 800/800 | 0/800 |

durability를 얻으니 처리량이 절반이 됐다.
비용의 출처는 예상과 달랐는데, 동기 Kafka append 자체는 hot path에서 5ms로 저렴했다.
느려진 건 Redis 연산들이었고, Kafka 컨슈머(projection)가 trading 안에서 같이 돌며 생긴 부하가 공유 자원 경쟁을 늘린 탓이다.
컨슈머를 별도 서비스로 분리하면 이 비용을 상당 부분 회수할 여지가 있다.

## 정리

동기 DB 쓰기(병목) → Redis 우선 + 비동기(2배, 유실 구멍) → durable 로그(유실 0, 처리량 절반)의 순서였다.
각 단계를 측정으로 열고 닫았다 — 병목은 76%, 1차 개선은 2배, 유실은 800건, 2차 복구는 0건, 비용은 절반.
"durable 하니 안 잃는다"는 durable 쪽만 재는 함정이라, 인메모리 쪽 유실(800건)까지 재현해 양면을 맞췄다.
