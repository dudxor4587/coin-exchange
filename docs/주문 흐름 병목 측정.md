# 주문 흐름 병목 측정

## 배경

MSA 분리를 끝내고 수평 확장(HPA)을 붙이기 전에, 무엇이 병목인지부터 측정했다.
병목을 모르고 HPA를 붙이면 엉뚱한 컴포넌트를 복제하게 된다.
특히 *"매칭엔진이 가장 부하가 심하니 거기를 확장해야 한다"* 는 통념이 사실인지부터 데이터로 확인할 필요가 있었다.

측정 환경은 단일 노트북(8코어) + OrbStack 위의 docker-compose(14개 컨테이너) + K6(200 VU)이다.
이 환경 자체가 최종 결론에 영향을 주는데, 그 부분은 마지막에 정리한다.

## 구간 분해

`OrderFlowService`의 매수 흐름 각 구간에 Micrometer Timer를 붙여 시간을 측정했다.

| 구간 | 평균 | 비중 |
|---|---|---|
| createOrder (DB insert) | ~2,900 ms | 76% |
| debit (HTTP → funds) | ~600 ms | ~20% |
| processMatches | ~140 ms | 4% |
| placeOrderBook (Redis) | ~5 ms | 0.1% |
| match (매칭 연산, Redis Lua) | ~4 ms | 0.1% |

매칭 연산 자체는 4ms로 전체의 0.1%였다.
*"매칭이 병목"* 이라는 통념은 데이터와 맞지 않았다.
시간의 대부분은 그 주변, 즉 DB 쓰기(createOrder)와 동기 RPC(debit)에서 발생했다.

이후 측정은 createOrder가 왜 2.9초나 걸리는지에 집중했다.
INSERT 한 줄이 2.9초일 수는 없으므로, 무언가를 기다리고 있다고 보는 게 타당했다.

## 가설 1 — 커넥션 풀 크기 (기각)

HikariCP 풀을 10 → 30 → 50으로 늘렸다.

| pool | throughput | createOrder |
|---|---|---|
| 10 | 56/s | 2857 ms |
| 30 | 55/s | 2562 ms |
| 50 | 51/s | 2410 ms |

처리량은 늘지 않았고 오히려 살짝 줄었다.
createOrder가 2857 → 2410으로 줄어든 것을 개선으로 볼 수도 있으나, 같은 조건 재측정의 변동폭이 2857~3473ms로 600ms씩 출렁였다.
300ms 차이는 변동폭 안이라 유의미한 신호로 볼 수 없다.
풀 크기는 throughput에도 createOrder에도 의미 있는 영향을 주지 않았다.

## 가설 2 — DB CPU 부족 (기각)

MySQL 컨테이너는 `cpus: 0.3`으로 묶여 있었다. 0.3 → 1.0 → 2.0으로 늘렸다.

| DB CPU | throughput | createOrder |
|---|---|---|
| 0.3 | 47/s | 3418 ms |
| 1.0 | 47/s | 3473 ms |
| 2.0 | 48/s | 3380 ms |

변화가 없었다. DB CPU는 병목이 아니다.

## 가설 3 — 디스크 I/O (기각)

여기서 *"동시 쓰기 I/O가 가상 디스크에서 직렬화된다"* 고 결론 내릴 뻔했으나, 디스크를 직접 측정한 적은 없었다.
DB 데이터를 tmpfs(RAM)에 올려 디스크 I/O를 제거하고 재측정했다.

| | createOrder | throughput |
|---|---|---|
| 디스크 | 3,400 ms | ~50/s |
| tmpfs (RAM) | 3,133 ms | 51.8/s |

RAM에 올려도 동일했다. 디스크는 병목이 아니다.

## 가설 4 — K6의 CPU 경쟁 (기각)

부하 도구(K6)와 시스템이 같은 노트북에서 도니, K6가 CPU를 점유해 시스템에 실제로 영향을 주는지 확인했다.

```
호스트 CPU: 63% idle
K6 프로세스: 3.4%
load average: 2.80 (8코어)
```

호스트는 63%가 유휴였고 K6는 3.4%만 썼다. CPU 경쟁이 아니다.

## 스레드 덤프

서버(DB)도 호스트도 유휴인데 처리량이 50/s에 막혔다.
자원이 남는데 처리량이 안 오르면, 스레드가 무언가를 기다리며 블록된 상태일 가능성이 높다.
부하 중 trading 서비스에 SIGQUIT을 보내 스레드 덤프를 떴다.

```
http-nio 워커 44개 중:
  41개 = DB 커넥션 대기 (HikariCP ConcurrentBag.borrow)
   2개 = funds HTTP 응답 대기
   1개 = 유휴
```

워커의 93%가 DB 커넥션을 기다리며 park 돼 있었다.
커넥션 풀(10)이 직접적인 대기 지점인 건 분명했다.
다만 가설 1에서 풀을 50으로 늘려도 throughput이 안 올랐으므로, 이 둘은 모순처럼 보인다.

## 서버는 모두 유휴였다

스레드는 커넥션을 기다리는데, 정작 그 커넥션을 쥔 동안 DB는 유휴였다.
부하 중 trading-db 상태를 조회하면 실행 중인 트랜잭션이 0개였다.

```
MySQL CPU: 11%
실행중 트랜잭션: 0개   (부하 중인데)
```

커넥션을 쥐고 있는 동안 DB 작업이 아니라 다른 것을 기다린다는 뜻이다.

Redis도 같은지 확인했다.

```
Redis CPU: 29.8%            (단일스레드인데 100% 아님)
처리 용량: 293,850 ops/sec  (실제 부하는 초당 수백)
blocked_clients: 0
```

trading에서 측정한 Redis 구간은 `placeOrderBook 602ms`, `match 384ms`로 높았다.
Redis 서버는 유휴(29%)인데 trading은 Redis 응답을 600ms 기다렸다. DB와 같은 패턴이다.

여기에 단서가 하나 더 있었다.

```
debitKrw (HTTP → funds)  = 10 ms   빠름
createOrder (JDBC → DB)  = 느림     (풀 10개에서 대기)
placeOrderBook (Redis)   = 느림     (Lettuce 커넥션 ~2개에서 대기)
```

HTTP만 빨랐다. HTTP 클라이언트는 커넥션이 넉넉해 대기가 없고, DB(풀 10)와 Redis(커넥션 ~2)는 작은 풀에서 스레드가 대기한다.
서버 처리 속도 문제가 아니라 *동시에 열 수 있는 커넥션 수 × 왕복 지연* 문제였다.

## 결론

측정 결과를 정리하면 다음과 같다.

```
- MySQL: CPU 11%, 트랜잭션 0     → 유휴
- Redis: CPU 29%, 용량 여유      → 유휴
- 호스트: 63% idle               → 유휴
- 워커 41/44 = 커넥션 대기
- 풀 크기 변경 무효 (병목이 DB → Redis로 이동)
- HTTP만 빠름 (커넥션 넉넉)
```

세 서버가 모두 유휴인데 스레드는 작은 커넥션 풀에서 대기한다.
풀을 넓히면 병목이 다음 풀(Redis)로 옮겨갈 뿐 throughput은 그대로다.
공통 원인은 모든 cross-container 왕복(JDBC·Redis·HTTP)이 통과하는 OrbStack 가상 네트워크의 동시 I/O 처리량이다.

> ~50/s는 아키텍처의 한계가 아니라 로컬 단일 노트북 가상화의 동시 I/O 한계다.
> 커넥션 풀은 스레드가 대기하는 지점이고, 원인은 그 너머의 가상 I/O 경로다.
> WAS를 묶지 않아 병목이 여기로 떨어졌고, 그래서 이 측정의 절대 처리량은 의미를 갖지 못한다.

## 측정 목적이 바뀌면 자원 제한도 바뀌어야 한다

저장소(MySQL/Redis)는 `cpus: 0.3`으로 묶었지만 WAS(trading/funds)는 제한을 걸지 않았다.
이 설정은 [매칭엔진 성능 비교](매칭엔진%20성능%20비교.md)에서 가져온 것이다.
당시 실험 변수는 *매칭엔진의 storage 종류*였고 WAS는 변수가 아니었으므로, WAS를 묶지 않은 것은 옳은 결정이었다.

그러나 이 측정의 목적은 다르다.
전체 주문 흐름의 병목을 찾고 수평 확장을 보려는 것이므로, *WAS 자체가 변수*다.
WAS가 무제한이면 병목이 통제 변수가 아니라 환경이 우연히 가진 한계(가상 네트워크)로 떨어진다.
목적이 바뀌었는데 자원 제한 설정을 그대로 물려받은 탓에, 병목이 측정 대상 바깥으로 샌 것이다.

수평 확장을 측정하려면 병목이 WAS 안에 있어야 한다.
WAS를 복제했을 때 늘어나는 자원이 처리량으로 이어져야 확장 효과가 드러나기 때문이다.
WAS에 CPU 제한을 걸면 병목이 WAS로 옮겨오고, 그때 복제 효과를 측정할 수 있다.

## 수평 확장 측정

kind(로컬 k8s)에 배포하고 trading 인스턴스 수를 1 → 2 → 3으로 늘리며 측정했다.
같은 측정을 두 조건으로 했다. trading CPU를 제한하지 않은 경우와, `0.3`으로 제한한 경우다.

| trading 인스턴스 | CPU 무제한 | CPU 0.3 제한 |
|---|---|---|
| 1대 | 91/s | 19/s |
| 2대 | 52/s | 48/s |
| 3대 | 59/s | 45/s |

두 조건의 방향이 정반대였다.

CPU 무제한일 때는 복제할수록 처리량이 떨어졌다(91 → 59/s).
trading이 병목이 아닌데 복제하니, 같은 공유 자원(DB·funds·가상 네트워크)을 더 많은 인스턴스가 두드릴 뿐이고 한 호스트의 자원 경쟁만 늘었다.
*병목이 아닌 컴포넌트를 복제하면 처리량은 오르지 않고 오히려 떨어진다.*

CPU를 `0.3`으로 제한해 trading을 병목으로 만들면 복제가 효과를 냈다(19 → 48/s, 약 2.5배).
다만 2대에서 처리량이 평탄해졌다(48 → 45/s).
trading 병목이 풀리자 병목이 공유 자원(funds·DB·가상 I/O)으로 옮겨갔기 때문이다.
앞에서 확인한 *가상 I/O 천장*이 trading을 풀어내자 새 병목으로 드러난 것이다.

별도 티켓팅 프로젝트에서 WAS를 `0.3`으로 묶고 1대 vs 3대를 측정했을 때 14.5/s → 91.1/s로 약 6배 증가했다.
그쪽은 WAS가 순수하게 CPU 병목이라 거의 선형으로 확장됐고, 우리는 병목이 곧 공유 자원으로 옮겨가 2대에서 평탄해졌다.
두 결과는 *복제가 효과를 내려면 그 컴포넌트가 실제 병목이어야 하고, 확장은 다음 병목에서 멈춘다*는 같은 법칙의 양면이다.

## 동기 I/O와 처리량의 관계

이 측정은 동기 I/O가 처리량을 어떻게 제한하는지 보여준다.

```
주문 1건 = 여러 I/O 왕복 (DB INSERT + Redis ×N + HTTP ×2)
처리량 ≈ (동시 커넥션 수) / (왕복 지연 × 왕복 횟수)
```

매칭 연산 자체는 4ms로 거의 비용이 없었지만, 그 앞뒤의 동기 I/O(주문 저장, 잔고 RPC)가 처리량을 결정했다.
우리 시스템에서 createOrder(동기 DB I/O)가 병목인 것은 hot path에 동기 I/O를 두는 비용을 그대로 보여준다.

## 정리

이 챕터의 측정을 순서대로 정리하면 다음과 같다.

1. 구간 분해 — 병목은 매칭(0.1%)이 아니라 동기 DB 쓰기(createOrder, 76%)였다.
2. 가설 기각 — 풀/CPU/디스크는 모두 무효였고, 절대 처리량은 가상 I/O 천장에 막혀 있었다.
3. 수평 확장 — trading이 병목일 때만 복제가 효과를 냈고(19 → 48/s), 곧 공유 자원 천장에서 멈췄다.

처리량을 끌어올리는 길은 *병목이 아닌 컴포넌트를 복제하는 것(HPA)*이 아니라 *병목 자체를 없애는 것*이었다.
우리 병목은 hot path의 동기 DB 쓰기이므로, 다음 작업은 주문 저장을 hot path에서 분리하는 구조 변경(Redis 우선 처리 + 비동기 영속화)이다.
그 작업은 별도 챕터에서 다룬다.
