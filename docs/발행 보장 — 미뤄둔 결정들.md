이 글은 [발행 보장(Transactional Outbox)](발행%20보장%20\(Transactional%20Outbox\).md) 챕터에서 의도적으로 미뤄둔 결정들을 정리한 글이다. <br>
"우리가 모르는 것을 안다"를 명시적으로 남겨두는 자리고, 6/7단계로 들어갈 때 그대로 todo 리스트가 된다. <br>
원칙은 단순하다. **측정 없이 도입하지 않는다**. 비용은 확정인데 이득은 추측인 거래는 안 한다. <br>

# 1. 다중 인스턴스 락 전략
지금은 인스턴스 1개에 릴레이 스레드 1개라 락 충돌 자체가 발생할 수 없다. <br>
funds-service와 trading-service의 `OutboxRepository`는 둘 다 `@Lock(PESSIMISTIC_WRITE)` 만 사용하고, 어느 락 전략을 써도 결과가 같다. <br>
다중 인스턴스로 운영해 본 적이 없으니 분산 환경에서 어떤 락 전략이 적합한지 답할 수 없는 상태다. <br>

6단계(k8s + HPA)에서 인스턴스를 늘리는 시점에 두 가지 옵션을 비교하게 될 것 같다. <br>
1. `SELECT ... FOR UPDATE NOWAIT` — 잠겨 있으면 즉시 실패하고 재시도
2. `SELECT ... FOR UPDATE SKIP LOCKED` — 잠긴 row를 건너뛰고 다음 row를 잡음

직관적으론 outbox 작업이 row 단위 순서가 무관하기 때문에 SKIP LOCKED가 더 잘 맞을 것 같지만, 측정 없이 단정하지 않는다. <br>
인스턴스 N개를 띄우고 같은 부하(예: K6 8000 req)에서 처리량과 인스턴스당 idle 시간, outbox 누적 깊이를 비교해서 결정한다. <br>

JPA 표준 hint로는 NOWAIT만 지원되고, SKIP LOCKED는 Hibernate 전용 hint나 native query가 필요하다는 점도 결정 비용에 포함된다. <br>

# 2. async Kafka publish
현재 릴레이의 발행 코드는 다음과 같다. <br>
```java
for (msg : pending) {
    kafkaTemplate.send(...).get();   // ← broker ack까지 동기 대기
    msg.markPublished();
}
```
`.get()` 으로 매 메시지마다 broker ack를 기다리고 있어서, Kafka producer가 내부적으로 batch 묶음 송신을 해주는 이득이 사라진다. <br>

이걸 다음과 같이 바꾸면 batch 이득을 살릴 수 있다. <br>
```java
List<CompletableFuture<?>> futures = pending.stream()
    .map(msg -> kafkaTemplate.send(msg.getTopic(), msg.getPartitionKey(), msg.getPayload()))
    .toList();
CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).get();
pending.forEach(OutboxMessage::markPublished);
```
다만 지금 바로 적용하지는 않았다. <br>
K6 throughput이 100/s인데, 이 수치가 정말 `send().get()` 때문인지 확신할 수 없기 때문이다. <br>
DB SELECT FOR UPDATE, 트랜잭션 commit, 직렬화, Hikari 풀 등 후보가 많고, 측정 없이 코드 복잡도(부분 실패 처리)를 늘리는 건 비용 확정 / 이득 추측인 거래다. <br>

먼저 drainBatch 메서드 안에서 시간이 어디 쓰이는지 분해해서 측정해보고, send가 압도적이면 그때 적용한다. <br>
부분 실패 처리는 각 future의 결과를 별도로 보고 실패한 메시지만 PENDING으로 남기는 방식이 자연스러운데, 7단계에서 retry topic 설계와 같이 묶일 자리다. <br>

# 3. 단일 릴레이 vs 다중 릴레이
인스턴스 1개에 릴레이 스레드 1개, push 신호 + 1초 타임아웃 폴링, 한 batch는 200건이라는 게 현재 구조다. <br>
이게 옳아서 단일 릴레이를 택한 게 아니라, 자기 자신과 충돌하던 두 스케줄러 구조를 가장 단순하게 회피한 형태일 뿐이다. <br>
최적이나 최종이 아니라는 점은 본문에도 적어두었다. <br>

처리량을 더 짜내야 한다면 다중 릴레이 옵션이 두 가지 있다. <br>
1. SKIP LOCKED 기반 — 락 전략으로 row 분배
2. id 기반 파티션 — `id % N == relayId`, 락 자체를 회피

다만 다중 릴레이로 가기 전에 위 (2)의 async Kafka publish가 먼저 풀어야 할 lever다. <br>
send 동기 호출이 throughput 천장을 만들고 있다면, 릴레이를 N개로 늘려도 같은 천장에 N번 부딪힐 뿐이기 때문이다. <br>
(2)를 적용하고도 throughput이 부족하면 그제서야 다중 릴레이 후보를 비교한다. <br>

# 4. retry topic / DLQ / 회로차단기
운영 단계의 내구성에 해당하는 항목들이다. <br>
- 컨슈머 처리 실패 시 retry topic으로 보내기 (지수 backoff)
- N회 실패 시 DLQ
- broker 자체 장애 시 publisher 측 회로차단기

본 챕터에선 atomicity 보장까지만 다뤘기 때문에, 이건 7단계 운영 이슈 챕터로 미뤘다. <br>

# 결론
> 미뤄둔 결정들은 모두 측정 없이 결정할 수 없는 것들이다. <br>
> 단일 인스턴스, 단일 릴레이, sync send는 현재 단계에서 가장 단순한 형태일 뿐, 최적이나 최종이 아니다. <br>
> 6/7단계에서 측정한 후, 이 doc의 항목들을 근거 있는 의사결정으로 바꿔 본문에 통합할 계획이다.
